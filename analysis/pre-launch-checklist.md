## STOP — Do These Before Any Real User Touches The System

### 1. Plain password — encrypt it, don't remove it

You want admins to be able to see what password they set for a student.
That's a valid need. But storing raw text in the DB is dangerous.

The fix: encrypt the display copy using Fernet (symmetric encryption).
Anyone who steals the DB sees garbage. The admin UI still shows the original password.

Install:
  pip install cryptography

Generate an encryption key (run once, save to .env):
  python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"

Add to .env:
  FERNET_KEY=your_generated_key_here

In db.py add two helpers:
  from cryptography.fernet import Fernet
  import os

  def _fernet():
      key = os.environ.get('FERNET_KEY')
      if not key:
          raise RuntimeError("FERNET_KEY not set in .env")
      return Fernet(key.encode())

  def encrypt_plain_password(plain: str) -> str:
      return _fernet().encrypt(plain.encode()).decode()

  def decrypt_plain_password(encrypted: str) -> str:
      return _fernet().decrypt(encrypted.encode()).decode()

Then in create_user() and update_user_password(), encrypt before saving:
  plain_password=encrypt_plain_password(plain_password) if plain_password else None

And when displaying in the admin UI, decrypt before showing:
  user['plain_password'] = decrypt_plain_password(user['plain_password'])

The login fallback (accepting plain_password at login) should still be REMOVED
because it creates a second authentication path that bypasses hashing.

### 3. SECRET_KEY — why the fallback is dangerous

config.py currently:
  SECRET_KEY = os.environ.get('SECRET_KEY') or 'mis-system-secret-key-change-in-production'

Flask uses SECRET_KEY to sign session cookies. The cookie contains your
user_id and role. If someone knows the key, they can forge a cookie that
says they are the superadmin — without ever logging in.

That fallback string is on GitHub right now. Anyone can forge admin sessions.

Fix:
  SECRET_KEY = os.environ.get('SECRET_KEY')
  if not SECRET_KEY:
      raise RuntimeError("SECRET_KEY must be set in .env — do not run without it")

### 4. Open redirect in /set-language route

auth.py set_language() does:
  next_url = request.args.get('next') or request.referrer
  return redirect(next_url)

An attacker can send a teacher a link like:
  yoursite.com/set-language/en?next=http://evil.com/fake-login

The browser follows it, the teacher sees a fake login page, types their password.

Fix — validate the redirect target stays on your domain:
  from urllib.parse import urlparse

  def is_safe_redirect(url):
      if not url:
          return False
      parsed = urlparse(url)
      # Allow only relative URLs (no scheme/host)
      return not parsed.scheme and not parsed.netloc

  # In set_language():
  next_url = request.args.get('next') or request.referrer
  if not is_safe_redirect(next_url):
      next_url = url_for('auth.dashboard') if 'user_id' in session else url_for('auth.login')
  return redirect(next_url)

### 5. plain_password returned in AJAX JSON responses

In admin.py edit_user_ajax() and add_user_ajax(), the JSON response includes:
  'plain_password': plain_pass if plain_pass else None

And:
  'plain_password': new_user.get('plain_password')

This sends the password (currently plain text) in every AJAX response to the browser.
It appears in browser developer tools, network logs, and any proxy between client and server.

Once you implement encryption (item 1 above), also decrypt before sending:
  'plain_password': db.decrypt_plain_password(user['plain_password']) if user.get('plain_password') else None

And make sure these AJAX responses are only sent over HTTPS in production.

### 6. Student number race condition

In admin.py, new student numbers are generated by finding the max existing number and adding 1.
If two admins add a student at the exact same millisecond, both get the same number,
one INSERT fails with a unique constraint error, and the admin sees a 500 error.

The system probably has one or two admins and this is unlikely in practice,
but fix it properly before launch with a PostgreSQL sequence:

  CREATE SEQUENCE student_number_seq START 1000;

Then in create_student:
  SELECT nextval('student_number_seq') as next_num

This is atomic — two simultaneous requests always get different numbers.

### 8. CSRF tokens (forms can be submitted by other websites)

Every POST form in your system (login, take attendance, submit grades, add student, etc.)
can be triggered from a malicious website while a user is logged in.

Example: A student is logged in, opens a malicious link, that page silently submits
a form to /teacher/attendance/take/5/3 with fabricated attendance data.

Fix — add Flask-WTF:
  pip install Flask-WTF

  # config.py — add:
  WTF_CSRF_ENABLED = True

  # app.py create_app() — add:
  from flask_wtf.csrf import CSRFProtect
  CSRFProtect(app)

  # Every HTML form — add this inside <form>:
  <input type="hidden" name="csrf_token" value="{{ csrf_token() }}">

  # AJAX POST calls — add header:
  fetch('/some/route', {
      method: 'POST',
      headers: { 'X-CSRFToken': '{{ csrf_token() }}' },
      body: formData
  })
